## 1. 背景与目标

### 1.1 背景（Why）

当前 MeterSphere（Vue3 + Arco Design）在日常测试协作中存在以下痛点：

| 痛点 | 现状 | 影响 |
|------|------|------|
| 详情入口不统一 | 多数列表仅支持点 ID/名称进详情，操作列首项多为「编辑」 | 用户习惯点「详情」查看，易误点编辑；跨模块体验不一致 |
| 测试规划偏「脑图」 | 测试计划详情「测试规划」Tab 为 KityMinder 思维导图 | 与团队既有「Markdown 测试计划文档」工作流脱节，难评审、难归档 |
| 项目切换辨识弱 | 顶栏项目下拉仅约 `50px` 宽，选中名称被 CSS 隐藏，靠 tooltip | 长项目名无法识别；当前组织不直观 |
| 测试报告外置 | 报告多在测试计划报告中心，用例模块无评审/用例 Tab | 用例侧缺少「一键出报告 + 可编辑」闭环 |
| Xmind 与执行用例混用 | 用例导入同时支持 Excel / Xmind，Xmind 导入会解析为功能用例 | 需要「只存文件、在线查看」的场景无法满足 |

### 1.2 目标（What）

1. **操作列统一增加「详情」**：位于「编辑」左侧，行为与点击 ID 进入详情一致；覆盖各业务模块列表。
2. **测试计划详情文档化**：「测试规划」更名为「测试计划」；删除思维导图模式，改为可编辑的 Markdown 测试计划模板（对齐《商城-订单改址-测试计划》结构）。
3. **顶栏项目切换可读**：下拉适配最长约 30 字项目名；组件左移；右侧展示当前组织名称。
4. **用例模块新增「测试报告」Tab**：格式对齐《测试报告-扫码模板…》样例；一键生成、资料自动取数、可编辑；不含「测试依据」、不含「六、附件」；缺陷双图 + 遗留风险自动汇总。
5. **用例 Tab 拆分**：下挂「执行用例」「Xmind用例」；执行用例保留现有能力但导入去掉 Xmind；Xmind用例独立存文件、可打开查看。

### 1.3 非目标（本期不做）

- 不改造接口测试 / 场景自动化报告体系本身（可复用能力，不合并入口）。
- 不把 Xmind用例 Tab 做成「导入后自动转功能用例」（与现有 Xmind 导入解析能力刻意拆开）。
- 不在本期实现测试报告「对外分享链接 / PDF 导出」除非评审明确要求（现有计划报告能力可作二期）。
- 不调整左侧一级菜单信息架构。

## 3. 需求分项方案

### 3.1 【P0】操作列新增「详情」按钮

#### 3.1.1 动机-行为-呈现

- **Why**：降低误入编辑；统一「先看后改」路径。
- **How**：列表行操作列在「编辑」左侧增加「详情」；点击后走与「点击 ID / 名称进详情」相同逻辑与权限。
- **What**：文案「详情」；权限与当前「查看详情」一致（通常为模块 `READ`，不要求 `UPDATE`）。

#### 3.1.2 覆盖范围（「所有模块」落地清单）

> **✅ 已确认**：覆盖范围**包含**系统设置 / 组织 / 项目管理等表格；凡存在操作列且存在详情入口、且操作列有「编辑」的列表，均需加「详情」。

| 模块 | 建议改动文件（前端） | 详情行为对齐 |
|------|----------------------|--------------|
| 测试用例-功能用例 | `caseTable.vue`（及计划内关联用例表若有编辑） | 同点 ID → 详情抽屉 |
| 测试用例-评审 | `caseReview/.../reviewTable.vue` | 同点 ID → 评审详情 |
| 测试计划 | `planTable.vue` | 同 `openDetail`；计划组按现有详情规则 |
| 缺陷 | `bug-management/index.vue` | 同点 ID → 缺陷详情抽屉 |
| 接口定义 / 接口用例 / 场景 | 对应 `apiTable.vue` / `caseTable.vue` / `scenarioTable.vue` | 同现有点 ID/名称进详情 |
| 测试计划报告列表、接口报告列表 | `reportList.vue` 等 | 若操作列有「编辑」则同样加「详情」 |
| 项目管理（成员、文件、环境、模板等） | `project-management/**` 中带操作列+详情的表格 | 同现有点 ID/名称/行进详情 |
| 组织设置 / 系统设置 | `setting/organization/**`、`setting/system/**` 中带操作列+详情的表格 | 同现有详情抽屉/页 |

**明确排除**：无详情入口的纯配置表；仅「更多」且无「编辑」的操作列；无「资源详情」语义的操作（如仅启用/禁用）。

#### 3.1.3 交互规则

1. 顺序：**详情 | 编辑 | …**（其后保持各模块原有按钮与「更多」）。
2. 无编辑权限时：仍可显示「详情」（有查看权限时）；无查看权限则整列按现有逻辑隐藏。
3. 宽度：操作列 `width` 按模块现有值上调约 40–60px，避免挤出。
4. i18n：优先复用已有 `*.detail` / 新增 `common.detail` =「详情」，中英繁统一。

#### 3.1.4 验收标准

- [ ] 上述清单模块操作列「详情」在「编辑」左侧。
- [ ] 点击「详情」与点击 ID 进入同一页面/抽屉、同一权限校验。
- [ ] 无编辑权限用户仍可点「详情」查看（在具备 READ 时）。

### 3.2 【P0】测试计划详情：「测试规划」→「测试计划」+ 富文本文档模板

#### 3.2.1 动机-行为-呈现

- **Why**：用结构化测试计划文档替代脑图，便于评审、归档、与外部文档对齐。
- **How**：Tab 文案变更；移除 `MsTestPlanMinder`；改为 **富文本（MsRichText）** + 标准章节模板；支持保存、导出、自动带出部分字段。
- **What**：Tab 名「测试计划」；内容为可编辑富文本文档，章节对齐样例文档；可导出。

#### 3.2.2 模板结构（对齐样例，字段可空）

以 `商城-订单改址-测试计划-20260717.md` 为章节骨架：

1. 文档信息（编号、版本、所属项目、测试模块、编制日期、编制人、审核人）
2. 项目背景
3. 测试目标
4. 测试范围（范围内 / 范围外）
5. 测试策略
6. 测试重点（主流程 / 规则与边界 / 权限与安全）
7. 测试环境与数据
8. 准入、暂停与退出标准
9. 测试进度
10. 人员与职责
11. 缺陷管理
12. 风险与应对
13. 测试交付物
14. 审批记录

**系统自动填充（首次生成或一键重置模板时）**：

| 字段 | 数据来源 |
|------|----------|
| 所属项目 | 当前项目名称 |
| 测试模块 / 文档标题建议 | 测试计划名称 |
| 编制日期 | 当天日期 |
| 编制人 | 当前用户名 |
| 文档编号 | 建议规则：`TP-{projectCode或缩写}-{planNum}-{yyyyMMdd}`（规则需评审确认） |

其余章节保留空表格 /「待填写」占位，由用户编辑。

#### 3.2.3 交互设计

1. 进入「测试计划」Tab：若该计划尚无文档内容 → 自动套用模板；若已有 → 展示已存内容。
2. 工具栏：**保存**（必选）、**导出**（必选：至少支持导出 Word 或 HTML/PDF 其一，具体格式实现阶段确定）、**预览**（建议）、**重置为模板**（二次确认，覆盖当前内容）。
3. 编辑器：复用 `MsRichText`（所见即所得富文本）；**不做**纯 Markdown 双态。
4. 权限：查看 `PROJECT_TEST_PLAN:READ`；编辑/保存/导出与计划编辑权限对齐（建议 `READ+UPDATE`；导出可仅需 READ）。
5. **删除思维导图与旧数据**：
   - 前端不再渲染 minder，无脑图入口。
   - **✅ 已确认**：当前系统无有效脑图业务数据，**旧 minder 数据一律删除处理**（含表数据清理 / 废弃接口下线，不做迁移提示、不保留只读回看）。

#### 3.2.4 数据与接口（建议）

| 项 | 建议 |
|----|------|
| 存储 | 新表或计划扩展字段：`test_plan_document`（plan_id、content HTML、update_user、update_time） |
| API | `GET /test-plan/{id}/document`、`POST /test-plan/{id}/document`（保存）、`GET /test-plan/{id}/document/export`（导出） |
| 旧数据 | 删除/废弃 minder 相关表数据与无用接口；首次进入一律套用富文本模板 |

#### 3.2.5 验收标准

- [ ] Tab 文案为「测试计划」（中英繁）。
- [ ] 页面无思维导图编辑器与保存脑图入口；旧 minder 数据已清理。
- [ ] 可见完整模板章节；富文本可编辑并持久化；刷新后内容保留。
- [ ] 支持导出；自动填充字段正确；重置模板需确认。

### 3.3 【P1】项目切换下拉加长 + 左移 + 右侧显示组织

#### 3.3.1 动机-行为-呈现

- **Why**：项目名可读；组织上下文可见，减少切错项目。
- **How**：加宽下拉并展示选中名称；在顶栏右区将其左移；右侧展示当前组织名。
- **What**：约 30 汉字可视宽度；组织名只读展示。

#### 3.3.2 布局方案（推荐）

当前：`Logo | TopMenu(居中) | [项目▼] 图标区…`（项目在最右前）。

**目标**：

```
Logo | TopMenu | …… | [项目名称下拉 ≈30字宽] [当前组织：xxx] | 通知/任务/帮助/…
```

- 「左移」含义：在 `right-side` 内，项目选择仍为该区域最左侧控件，但其**可视主体变为项目名**（不再是 50px 箭头），视觉重心左移；组织文本紧随其后。
- 不占用 Logo 区；不改变 TopMenu 居中逻辑（避免大改导航栅格）。

> **✅ 已确认**：采用本方案（右区加宽），**不**将项目选择移到 Logo 旁；不设备选方案 B。

#### 3.3.3 规格

| 项 | 规格 |
|----|------|
| 下拉可视宽度 | 按中文字号约 14px，30 字 ≈ `30 * 14 ≈ 420px`；建议 `min-width: 200px; max-width: 420px`（或固定 `w-[420px]`） |
| 展示 | 取消隐藏 `.arco-select-view-value`；超长省略号 + tooltip 全文 |
| 组织展示 | 文案：「当前组织」+ `orgList` 中匹配 `currentOrgId` 的 `name`；超长省略 |
| 交互 | 组织名本期只读（切换组织仍走现有组织入口） |
| 响应式 | 小屏可缩短为 `max-width: 240px`，组织名可仅图标+tooltip |

改动主文件：`frontend/src/components/pure/navbar/index.vue`；样式与 i18n（`settings.navbar.*`）。

#### 3.3.4 验收标准

- [ ] 30 字项目名主体可见（不依赖 hover 才知道当前项目）。
- [ ] 项目下拉右侧可见当前组织名称。
- [ ] 切换项目行为与现网一致。

### 3.4 【P0】测试用例模块 —「测试报告」Tab

#### 3.4.1 动机-行为-呈现

- **Why**：在用例模块直接产出可编辑测试报告，对齐团队报告模板。
- **How**：在「评审」右侧增加顶栏 Tab「测试报告」；提供报告列表 + 详情编辑；「一键生成」从系统取数填充。
- **What**：报告结构见下；支持二次编辑与保存。

#### 3.4.2 信息架构

```
测试用例模块顶栏：
  用例 | 评审 | 测试报告（新增）
```

路由建议：`/case-management/testReport`，`meta.isTopMenu: true`，locale：`menu.caseManagement.testReport` =「测试报告」。

页面结构建议：

1. **列表页**：报告名称、关联范围（版本/计划/时间）、生成人、更新时间、操作（详情｜编辑｜删除｜…）。
2. **详情/编辑页**：按模板分节渲染；支持编辑模式。

#### 3.4.3 报告输出格式（对齐样例，并按需求裁剪）

| 章节 | 是否输出 | 内容来源 / 规则 |
|------|----------|-----------------|
| 一、版本概览 | ✅ | 产品/系统名称、产品经理、开发负责人、测试负责人、计划测试周期、实际测试周期、上线计划日期 → **可编辑**；生成时带出**所绑定测试计划**名称与项目名，当前用户为测试负责人默认值 |
| 二、测试范围与目的 | ✅ 仅「1. 测试内容」 | 可编辑文本；**不输出「2. 测试依据」**（系统无记录） |
| 三、质量评估 | ✅ | 见下 |
| 　3.1 测试执行统计 | ✅ | 按**所绑定测试计划**汇总功能用例执行：总数/通过/失败/阻塞/执行率/通过率；说明文案可写死与样例一致 |
| 　3.2 缺陷统计与分析 | ✅ | **图1**：按「处理人 × 处理状态」维度柱状图；**图2**：按「缺陷类型」饼图 |
| 四、遗留问题与风险 | ✅ | **自动**：汇总执行结果为「阻塞」「失败」的用例（编号+名称+结果）；用户可再手工补充说明 |
| 五、测试结论与建议 | ✅ | 结论、建议 → 可编辑；生成时可给默认结论草稿（如通过率=100% 则「测试通过」） |
| 六、附件 | ❌ | **删除整节**（不展示用例/缺陷外链占位） |
| 页脚 | ✅ | 报告撰写人、日期（默认当前用户与当天） |

> 通过率计算说明与样例对齐：`(通过数 / (用例总数 - 阻塞数 - 失败数)) * 100%`（需在 UI 旁注释；若分母为 0 显示 `-`）。

#### 3.4.4 「一键生成报告」流程

> **✅ 已确认数据范围**：报告**绑定测试计划**。生成前需先选择一个测试计划，所有统计（执行统计、缺陷、遗留风险）均基于该计划的关联用例与缺陷。

```
用户点击「一键生成报告」
  → 选择所绑定的测试计划（必选）
  → 后端聚合：该计划内用例执行统计、关联缺陷统计、失败/阻塞用例列表
  → 前端套模板生成可编辑报告草稿并落库（记录 planId）
  → 进入编辑页，用户改文字/数字后保存
```

**自动取数映射（建议）**：

| 报告块 | 系统数据 |
|--------|----------|
| 执行统计 | **所绑定测试计划内**功能用例执行结果汇总 |
| 图1 柱状图 | 计划关联缺陷：处理人（**`handleUser`**）× 状态（status）计数 |
| 图2 饼图 | 计划关联缺陷：按新增字段 **「缺陷类型」** 计数 |
| 遗留风险 | 计划内 `lastExecResult ∈ {BLOCKED, ERROR/FAILED}` 的用例列表 |

图表：前端可用项目已有图表组件（工作台缺陷图可参考 `workbench/.../defectMemberBar.vue` 等）或 ECharts。

#### 3.4.5 编辑与权限

- **✅ 已确认：生成后全文可编辑，含统计表数字**。生成时以计划数据填充为「初始草稿」，用户可手工修改任意文字与数字并保存（保存后以用户编辑值为准，快照与实时数据脱钩）。
- 提供「刷新统计」按钮：重新拉取所绑定计划的实时数据覆盖统计块（二次确认，避免覆盖手工修改）。
- 图表数据随统计快照生成；数字被手工改动后，图表按快照值重绘（或提示「已手工修改，刷新以同步」）。
- 权限建议：新权限点 `FUNCTIONAL_REPORT:READ / READ+ADD / READ+UPDATE / READ+DELETE`，或先复用用例模块权限以降成本（评审确认）。

#### 3.4.6 缺陷字段增强（支撑图2与可配置枚举）【本期新增范围】

| 与功能用例关系 | **不自动解析为用例**；用户需要结构化用例时仍用「执行用例」的 Excel 导入或手工新建 |

## 6. 风险与注意
