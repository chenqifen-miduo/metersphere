## 1. 背景与目标

### 1.1 Why（动机）

| 痛点 | 影响 |
|------|------|
| 编辑后忘记点「保存」 | 切页 / 关抽屉后内容丢失 |
| 保存按钮常在页顶或页底 | 长表单需反复滚动，操作成本高 |
| 误改后缺少可靠回退 | 只能凭记忆重填，或依赖运维恢复 |

### 1.2 How / What（目标摘要）

| # | 需求 | 优先级 |
|---|------|--------|
| 1 | 可编辑详情：**失焦保存** + **内容变更防抖自动保存** | P0 |
| 2 | 自动/手动保存失败：**Toast** + **阻断离开**（页 / Tab / 抽屉） | P0 |
| 3 | **保留**显式「保存」按钮；**Ctrl+S** 触发同手动保存 | P0 |
| 4 | **整单 Undo 栈**（最多 **2** 步）；自动保存成功后仍可撤；**跨天**有效 | P0 |
| 5 | **Ctrl+Z** / **Ctrl+Shift+Z**（Undo / Redo） | P0 |
| 6 | 编辑态**加锁**；他人打开只读并提示「xxx 正在编辑」；**15 分钟**无操作自动释锁 | P0 |
| 7 | 无编辑权限：只读 + 无自动保存 | P0 |
| 8 | 状态提示：保存中 / 已保存 / 保存失败 / 有未保存更改（取产品默认态） | P1 |
| 9 | 全产品统一能力；**分期接入**（首期三模块，其余复用 SDK） | P0 |

### 1.3 非目标（本期不做）

- 无限历史版本 / 完整审计 diff 浏览（仅滚动保留约 2～3 份快照）。
- 字段级 Undo、附件上传/删除纳入这 2 步 Undo。
- Agent / Excel / 批量路径的**自动保存**（这些路径一次性写入正式数据）。
- 管理员「强制抢锁」（首期不做；他人仅只读等待或稍后重试）。
- 离线优先的完整 PWA 同步（本地草稿仅作网络失败弱兜底）。

### 1.4 路径策略（已确认）

| 路径 | 自动保存 | Undo（最多 2 步） |
|------|----------|------------------|
| 人工表单 / 富文本编辑 | ✅ | ✅（每次成功保存推入快照） |
| Agent 写入 | ❌ | 可选：整次写入记 **1** 个快照（首期可后置） |
| Excel / Xmind 导入 | ❌ | 可选：整次导入记 **1** 个快照（首期可后置） |
| 批量编辑 | ❌ | 批量提交成功后记 **1** 个快照（首期可后置） |

### 1.5 「整单」边界（已确认）

- **计入 Undo**：同一资源上的基本信息、步骤/正文、自定义字段等业务正文，一次成功保存 = 栈上一步。
- **不计入 Undo**：附件上传/删除、评论、关联关系等（避免与文件域耦合；后续可单独立项）。

## 2. 产品行为规格

### 2.1 自动保存

| 项 | 规格 |
|----|------|
| 触发 | ① 表单控件 / 富文本 **失焦**且存在未保存变更；② 内容变更后 **防抖 1.5～2s**（已确认 可接受） |
| 成功 | 更新正式数据；推入版本快照（见 §3）；状态 →「已保存」 |
| 失败 | Toast 错误；状态 →「保存失败」；**禁止**路由离开 / 关抽屉 / 切会丢上下文的 Tab（二次确认仍可强制离开并提示数据可能丢失） |
| 与手动保存 | 同一套保存 API；按钮与 Ctrl+S 显式调用，不取消自动保存 |

### 2.2 撤销 / 重做

| 项 | 规格 |
|----|------|
| 模型 | 服务端滚动快照 + 客户端当前指针；**最多可回退 2 步** |
| 跨天 | 快照落库，换设备/刷新后仍可 Undo（在锁与权限允许下） |
| Undo | 恢复到上一快照覆盖正式数据，并刷新 UI；可再 Undo 至多共 2 次 |
| Redo | Ctrl+Shift+Z，在 Undo 后沿快照链前进 |
| 新编辑 | 在非栈顶产生新的成功保存时，丢弃「当前之后」的 redo 链 |

### 2.3 编辑锁

| 项 | 规格 |
|----|------|
| 获取 | 进入可编辑详情且有写权限时尝试加锁 |
| 持有 | 心跳或操作续期；**15 分钟无操作**自动释放 |
| 冲突 | 他人进入：只读 + 文案「{用户名} 正在编辑」 |
| 释放 | 离开页 / 关抽屉 / 主动退出编辑 / 超时 |

### 2.4 状态呈现（默认建议）

编辑区附近轻量状态：

- `idle`：无未保存变更（可显示「已保存 · HH:mm」）
- `dirty`：有未保存更改
- `saving`：保存中…
- `error`：保存失败（可点「重试」）
- `locked-readonly`：他人编辑中

## 3. 技术方案要点

### 3.1 总体建议（已确认方向）

**后端短版本快照为主，前端本地草稿仅弱兜底。**

```text
正式表（始终 = 最新成功保存态）
    │
    ├── resource_edit_lock（编辑锁）
    └── resource_edit_snapshot（滚动快照 v-1 / v-2，供 Undo）
前端 LocalStorage：仅「尚未成功落库」的失败暂存，恢复后优先 flush
```

不采用日常「草稿表与正式表双写」：避免一致性问题；Undo 读快照回写正式表即可。

### 3.2 数据模型（草案）

#### `resource_edit_lock`

| 字段 | 说明 |
|------|------|
| `id` | PK |
| `resource_type` | 如 `FUNCTIONAL_CASE` / `BUG` / `TEST_PLAN_DOCUMENT` |
| `resource_id` | 资源 ID |
| `project_id` | 项目 |
| `holder_user_id` | 持锁人 |
| `holder_user_name` | 展示用（可冗余） |
| `expire_time` | 过期时间（续期刷新） |
| `create_time` / `update_time` | |

唯一索引：`(resource_type, resource_id)`。

#### `resource_edit_snapshot`

| 字段 | 说明 |
|------|------|
| `id` | PK |
| `resource_type` / `resource_id` / `project_id` | 同上 |
| `seq` | 单调序号 |
| `payload` | JSON（整单可序列化正文；大字段可压缩） |
| `content_hash` | 可选，防重复快照 |
| `create_user` / `create_time` | |

策略：每资源最多保留 **3** 条（当前正式态可由正式表表达，快照保留最近 2 次成功保存前的状态，或保留最近 2 次成功点——实现时取「Undo 两次」可还原即可）。定期清理过期快照（如 30 天）。

### 3.3 API（草案）

| 方法 | 路径意向 | 说明 |
|------|----------|------|
| POST | `/resource-edit/lock/acquire` | 加锁；失败返回持锁人信息 |
| POST | `/resource-edit/lock/heartbeat` | 续期（建议 1～2 分钟一次） |
| POST | `/resource-edit/lock/release` | 释锁 |
| POST | `/{业务}/save`（复用或包装现有更新） | 保存成功后写快照 |
| POST | `/resource-edit/undo` | 整单回退一步 |
| POST | `/resource-edit/redo` | 重做一步 |
| GET | `/resource-edit/snapshot/meta` | 可 Undo/Redo 步数 |

各业务保存仍走既有校验与权限；快照与锁为横切能力。

### 3.4 前端 SDK（建议）

统一 composable / 组件包装，例如 `useAutoSaveEditor`：

- 绑定 dirty / saving / error
- 防抖 + 失焦
- beforeRouteLeave / 抽屉 `before-close` 拦截
- 快捷键注册（仅当前激活编辑器）
- 加锁心跳与只读降级

首期业务只「接入」，不各自实现一套。

### 3.5 分期（已确认）

| 阶段 | 范围 | 说明 |
|------|------|------|
| **S1 基建** | 锁表、快照表、通用 API、前端 SDK、状态条组件 | 可先用功能用例联调 |
| **S2 试点** | ① 功能用例详情 ② 缺陷详情 ③ 测试计划「测试计划」Tab | P0 验收主战场 |
| **S3 推广** | 用例评审、计划其他可编辑区、项目设置等 | 按同一 SDK 接入 |
| **S4 增强** | Agent/导入/批量成功后的单次快照（可选） | 非首期阻塞 |

**预估工期（粗）**：S1 **5～8** 人日；S2 **8～12** 人日；S3 按模块 **2～4** 人日/块。待技术负责人按代码现状修正。

## 4. 验收清单（首期）

| 编号 | 场景 | 期望 |
|------|------|------|
| T1 | 编辑用例后不点保存，失焦或 2s 内 | 自动保存成功，刷新后内容仍在 |
| T2 | 保存失败（模拟网络错误）后离开 | Toast + 拦截确认 |
| T3 | 手动保存 / Ctrl+S | 与自动保存等价成功 |
| T4 | 连续成功保存 3 次后 Ctrl+Z 两次 | 回到对应历史正文；再 Z 无效或提示 |
| T5 | Undo 后 Ctrl+Shift+Z | Redo 恢复 |
| T6 | 刷新页面后仍可 Undo（未超快照保留） | 跨会话 / 跨天成立 |
| T7 | 用户 A 编辑中，用户 B 打开 | B 只读 +「A 正在编辑」 |
| T8 | A 15 分钟无操作 | 锁释放，B 可编辑 |
| T9 | 无编辑权限用户 | 只读，无自动保存、无锁抢占 |
| T10 | 上传附件后 Undo | 正文回退，**附件不随这 2 步回滚** |
| T11 | Agent/Excel/批量 | 无自动保存；不因防抖产生中间态 |

## 5. 风险与人工审核点

| 风险 | 说明 | 建议 |
|------|------|------|
| 富文本大字段快照体积 | payload 膨胀 | 压缩 / 仅存变更块；限制单条大小 |
| 与枢纽同步（默认项目）叠加 | 用例自动保存触发频繁 sync | 同步侧防抖或 hash 短路（已有 content_hash 思路） |
| 加锁误伤 | 浏览器崩溃未释锁 | 严格过期时间 + 心跳 |
| 全产品一次性改造 | 回归面过大 | **必须**按 S2 试点再推广 |
| 安全 | 快照含敏感正文 | 与资源同权限校验；禁止越权读快照 |

**需人工审核**：快照 JSON 字段白名单、权限模型、与现有 `update` 接口事务边界。
